FinanceBot — Product Requirements Document (Build Reference) v0.8
Derived from PRD v0.7 (June 29, 2026). Incorporates resolutions to Jan Bena & Jose Pizarro's review comments (July 7, 2026) and subsequent corrections. Acceptance criteria are condensed into each user story's key requirements; Requirement IDs are preserved for traceability.
1. Project Overview
FinanceBot is a course-tuned, AI-powered practice chatbot for UBC Sauder finance courses. It delivers multiple choice (4-option) and True/False practice questions with adaptive explanations, grounded strictly in instructor-provided course materials via RAG. Students practice at their own pace, receive immediate feedback, and identify gaps before quizzes and exams.
The platform is multi-course from the outset: each course is independently configured (materials, hierarchy, exam structure, feedback behaviour, question-generation guidance).
Pilot: COMM 298 (Introduction to Finance), UBC Sauder, Winter Term 1 2026/27 (Sept–Dec 2026), ~180 students. Scale targets: COMM 370 (~120 students) and COMM 371 (~80 students) in Winter Term 2; platform designed for 800+ students/year. Pilot materials include lecture notes, instructor narrative scripts, practice problem sets, practice midterm/final exams, and existing parameterized question scripts.
Scope & delivery approach. This PRD defines the full target feature set for the platform, not a stripped-down pilot subset — the product's value depends on the instructor-side content pipeline and the student-side adaptive engine working together (see the product-definition note at the end of §5.1). Build sequencing against the TLEF milestones (Apr–Aug design, Aug–Sep internal testing, Sep–Dec pilot deployment, Dec interim evaluation, Jan–Mar refinement and WT2 expansion) is maintained separately in the Phasing Document, so that this PRD and the phasing plan don't duplicate or drift from each other. See §11 for the risks this scope creates and how they're mitigated.
2. Technology Stack
Languages & Runtime
TypeScript end-to-end: strict mode everywhere, with shared compiler options in a common tsconfig.base.json that the client and server each extend. The backend compiles to CommonJS; the client is authored as native ES modules. Shared domain types (Question, option roles, mastery states, exam templates) live in a shared types location on the server.
Node.js 18+: single runtime for the whole system. Instructor-authored parameterized-question scripts are JavaScript and run on this same runtime.
Frontend: Plain TypeScript Client
Plain TypeScript, HTML, CSS, compiled by tsc to static JavaScript, served directly by the Express server. FinanceBot's role-separated surfaces (student practice, instructor dashboard, TA queue, admin console) are views within this client app.
KaTeX + Markdown rendering for formulas, tables, and simple visuals (ST-P03 AC4). Chart library (e.g., Chart.js) for instructor/admin analytics visualizations.
Backend: Express API Service
A single Node.js + Express service serves both the static client and all /api routes. Routes delegate to services, which compose isolated integration components.
Passport.js + passport-ubcshib + express-session for CWL SSO via UBC Shibboleth; sessions in MongoDB (connect-mongo); no local credentials anywhere. API hardening: helmet, cors, rate limiting, request validation. multer for material uploads.
Notifications: in-app only, delivered via polling. No email channel in MVP — see §4.3 for how notification priority and batching prevent per-event noise without an email channel.
Data Layer
MongoDB (native driver): primary datastore. Indicative models: User, Course, Theme, LearningObjective, Question, QuestionVersion, AttemptRecord, Material, MasteryProfile, ReviewBookEntry, ExamTemplate, ExamAttempt, Flag, Notification, AuditLog. (QuestionVersion and AttemptRecord are new relative to v0.7 — see Data Model & Long-Term Operations below; they underpin versioning, mastery calculation, and flag remediation.)
Sessions in MongoDB via connect-mongo. MongoDB-backed background jobs (e.g., Agenda) handle generation pipeline runs, batch pre-seeding, term-expiry sweeps, mastery evaluation batches, difficulty recalibration, and research-export anonymization.
AI / RAG Pipeline
ubc-genai-toolkit-* (core, llm, embeddings, chunking, document-parsing) is the retrieval and generation foundation. All toolkit packages are pre-1.0: pin exact versions and budget for compatibility shims.
LLM access via the toolkit's provider abstraction: LLM_PROVIDER / model env configuration (ollama | openai | anthropic | ubc-llm-sandbox), supporting AD-07's requirement that each pipeline step has an independently assignable model. Pipeline steps: generator, structure validator, reviewer, and mastery evaluator — the mastery evaluator is a separate, independently-configurable step from the question-generation trio, and is expected to run on a cheaper/faster model since it performs structured classification over short inputs rather than grounded generation (see §9.2).
Qdrant vector database, scoped per course. OpenAI embedding model as deployed default, env-configurable. Document ingestion via the toolkit's document-parsing module (PDF, DOCX, PPTX, HTML, Markdown); plain-text and public-URL ingestion are thin application-side steps.
Parameterized question execution: instructor-authored JavaScript generate() functions, PrairieLearn convention, executed in Node worker_threads with resource limits, no network access, no filesystem writes, seeded randomization.
Testing
Jest + ts-jest + supertest for unit/integration tests, including pipeline-state transitions (Draft → Pending Review → TA/Instructor-reviewed → Approved), permission-gating, auth-gated endpoints. Playwright for a critical-path E2E suite (login → practice → feedback → Review Book; exam-mode integrity). @axe-core/playwright for WCAG scans (accessibility target formally WCAG 2.1 AA — see System Requirements below).
Infrastructure & Deployment
Single deployed Express process; horizontal scaling for concurrency peaks (AD-04). Environment-driven configuration in one typed config module: MONGODB_URI/MONGODB_DB_NAME, SESSION_SECRET, QDRANT_URL/QDRANT_API_KEY/QDRANT_COLLECTION, SAML_*, LLM_PROVIDER/LLM_DEFAULT_MODEL/LLM_ENDPOINT/LLM_API_KEY with per-pipeline-step model selection (generator/validator/reviewer/mastery evaluator), EMBEDDINGS_PROVIDER/EMBEDDINGS_MODEL, admin CWL allowlist, worker limits for parameterized-question execution. Local dev backing services via Docker (MongoDB, SAML IdP, Qdrant), mirroring staging/production. Feature flags as platform settings in MongoDB (AD-07). CWL integration requires PIA/DAR through UBC IAM before go-live (§11).
Data Model & Long-Term Operations
Entity relationships. Course is the top-level container, owning its own Theme → LearningObjective hierarchy, Materials, registration code/term dates, and an independent Question Bank — course data is isolated by default. Material↔Theme/LO and Question↔LO/Theme are both many-to-many. Question is not a single mutable record: every edit produces a new QuestionVersion; the Question record points at the current version, but every prior version is retained permanently so historical answers remain interpretable.
AttemptRecord is the central hub of the data model — an independent entity, not a sub-document of Question. Every student answer creates one, pinning: User, the specific QuestionVersion served (not just the Question), the served LO/Theme context, mode (Topic Practice / Review Book / Exam Prep), the feedback strategy active at that moment, and any randomized parameter values shown. MasteryProfile is a (User, LO) rollup computed from AttemptRecords, not a store of raw judgments. ReviewBookEntry references a Question plus the AttemptRecord that triggered it. ExamAttempt groups a batch of AttemptRecords (mode = Exam Prep) under an ExamTemplate instance. Flag attaches to a specific QuestionVersion (not the Question generally), so remediation can identify precisely which version was implicated. Notification and AuditLog are cross-cutting: Notification references its triggering event (flag, auto-pause, review-backlog threshold); AuditLog references the acting User and the target entity, covering all reviewer/instructor/TA actions. User is keyed by CWL PUID, carrying per-course role and any Admin capability overrides.
Storage & long-term accessibility. All of the above lives in the MongoDB collections already defined above — no additional storage system. Post-expiry, course data is not deleted or moved; access simply narrows to instructor read-only (IN-A05). Long-term interpretability is guaranteed by the version model itself: content changes over time, but every historical AttemptRecord is pinned to the exact version it was served against, so old records remain correctly interpretable even after later edits, LO merges, or splits.
How the system is built: fully specified above (Node.js + Express + MongoDB + Qdrant + ubc-genai-toolkit RAG pipeline); this subsection only adds the data-relationship layer on top, it doesn't change the technology choices.
How it's maintained — three concrete practices:
Before switching the production LLM/embedding provider, run a regression pass over a fixed set of known-good Approved questions through the reviewer agent and confirm pass rate hasn't regressed, before shifting production traffic.
Embedding-model changes (which force a Qdrant collection rebuild) are scheduled only at term boundaries, never mid-term, to avoid a retrieval gap interrupting active student practice.
Schema changes (new fields, enum changes) ship with a migration script; the system does not assume old and new-format data can't coexist mid-term.
Two open items, not resolved here — flagged for a non-product decision:
Backup/disaster-recovery policy (frequency, RPO/RTO) — not addressed anywhere in this document; an infra/budget decision, to be settled with UBC IT.
Post-pilot technical ownership of the codebase/system itself (distinct from the Admin role, which is already assigned to Sauder Learning Services post-pilot per §3) — flagged as TBD, to be resolved before Winter Term 2 scaling.
System Requirements
Response time: page load (course home, question view) <2s on campus network, time-to-interactive <3s. Question serving and answer→feedback round-trip <500ms (pure DB reads on this path — no LLM call sits in the student-facing critical path, since only Approved, pre-reviewed content is ever served). Instructor-side generation/regeneration (IN-Q11/IN-Q12, which do call the LLM) targets <15s with a visible loading state.
Concurrency: ≥250 concurrent active sessions without degradation for the pilot (180 enrolled + buffer); p95 <1s on read APIs at that load. WT2 validation target: 500 concurrent sessions, load-tested before WT2 enrollment opens.
Accessibility & compatibility: WCAG 2.1 AA (formalizes what the existing @axe-core/playwright test suite already scans for). Evergreen-browser support (latest two versions of Chrome/Firefox/Safari/Edge). Desktop is the primary optimization target (this is a sustained-use study tool — tables, formulas, long explanations); mobile/tablet must be usable end-to-end but isn't a dedicated design investment in MVP.
Monitoring/alerting: out of scope for the application itself — this is operational infrastructure (uptime checks, alert delivery), not a FinanceBot feature, and is handled by separate ops tooling outside this PRD.
Explicitly Out of Scope (MVP)
LTI / Canvas API / grade passback — stretch goal, not MVP.
Local password auth — CWL only.
Email notifications — no email channel in MVP at all. All notifications (including auto-pause alerts and the daily activity summary, §4.3) are in-app only; no SMTP or mailer dependencies.
3. Roles
Student — practices questions by topic, takes simulated exams, revisits mistakes via a personal Review Book. Feedback is immediate and one-directional in MVP (see §5.1).
Instructor — sets up the course, validates AI-generated questions, configures exams and feedback behaviour, monitors class and individual performance.
TA — reviews questions and marks them ready for instructor sign-off, and triages student flags to lighten instructor load. Never has final approval authority.
Admin — platform operations role (not teaching): manages accounts, roles, and permissions across courses; monitors platform health. Held by LTIC for the pilot, transferring to Sauder Learning Services post-pilot (technical/system ownership is separate — see Data Model & Long-Term Operations above).
4. Common Components
4.1 Authentication & Course Access
All users authenticate via CWL (UBC single sign-on). There is no separate FinanceBot account and no passwords are ever stored or displayed. Each user is identified by their CWL PUID, which links their mastery profile, answer history, Review Book, and enrollments across sessions and devices. No PII beyond what CWL provides at login is stored.
Per-student, per-course data retained: answer history (question, option selected, correct/incorrect, timestamp), mastery profile (per-LO status and progression), Review Book entries (auto-collected and bookmarked), skip records, and session activity (questions attempted, time spent). Data persists after course expiry for research and course improvement, subject to the consent model below.
Consent (two tiers, governed by UBC Policy — not directly by BC FIPPA):
Mandatory, service-necessary processing: the data listed above, required for the app to function (answer history, mastery profile, Review Book), is covered by a mandatory acknowledgement in the guided onboarding flow (alongside the copyright disclaimer, below). This isn't optional — the service can't function without it.
Optional, research-export consent: a separate, declinable consent covers whether a student's de-identified data may be included in research exports (§6.4/IN-A05). Declining doesn't affect normal practice access — it only excludes that student's data from de-identified research exports.
Students enroll with an instructor-provided registration code, cross-checked against an instructor-maintained roster — the code alone does not grant access; the student's CWL identity must also appear on the course roster (uploaded/maintained by the instructor). This closes the gap where a shared code alone can't verify enrollment. Access expires automatically at the instructor-set term end date; instructors can extend access for individual students (e.g., exam deferrals). Direct enrollment via UBC Academic API and Canvas integration remain stretch goals (Academic API would eventually remove the need for manual roster maintenance).
All displayed content is copyrighted (instructor / UBC Sauder). A copyright disclaimer is shown in a guided onboarding flow on first login, requiring acknowledgement; within practice, the disclaimer appears near the submit button on each question view. Each question and feedback view shows the student's CWL username as a lightweight watermark (screenshot deterrent).
CWL integration requires a Privacy Impact Assessment (PIA) and Data Access Request through UBC Identity and Access Management before go-live (§11). Instructor/TA access is granted via admin interface or by the course instructor; TA course pairings are independent (removal from one course doesn't affect another). Admins have cross-course visibility.
ST-E01 — CWL login. Landing screen has a single "Log in with CWL" control (no email/password fields, no account creation). Successful auth returns the user to their role-appropriate home; first login maps the CWL PUID to a FinanceBot identity with no profile-creation step; the same PUID resolves to the same identity thereafter. Failed/cancelled auth returns to the landing screen with a retry message and no partial session.
ST-E02 — Course enrollment by code + roster check. An "Add a course" control opens a single text input with a "Join course" button (disabled while empty). Valid code + roster match: course added immediately, navigate to course home. Valid code but not on roster: distinct "you're not on the roster for this course — contact your instructor" message. Invalid code: inline "code not recognized" error. Expired code: distinct "this course has ended" message. Duplicate code: no duplicate record, informational message. Expired courses remain visible in the course list in an inactive state; opening one shows a course-ended message.
ST-E03 — Identity-scoped activity. All practice activity, mastery, and Review Book entries are recorded per PUID per course. Re-login restores prior state exactly; multi-course progress never blends. Interrupted sessions resume with the persisted mastery state (at most a single unsubmitted question may be discarded). After term end, the student's history and Review Book are read-only; no new practice can start.
4.2 Roles & Capability Model
Permissions are configuration, not code. Every capability (approve/mark-reviewed questions, handle flags, view analytics, configure exams, manage TAs, etc.) is independently assignable to a role. Admins set platform defaults and adjust assignments without code changes.
4.3 Question Flag Loop
Students flag suspected wrong questions in one click without interrupting practice. Flagged questions stay active until the instructor resolves them — unless the flag count reaches the configured auto-pause threshold, at which point the question is automatically pulled from active use pending review.
Auto-pause threshold: a question auto-pauses when it meets either of two independent conditions — (a) at least 5 students have attempted it and at least 30% of them flagged it, protecting against small-sample noise; or (b) flag count reaches 15 in absolute terms regardless of percentage, protecting against a real problem being diluted in a large denominator (e.g., 20 flags out of 100 attempts is a genuine issue even at only 20%). Both the count and percentage thresholds are instructor-configurable in Course Settings.
Notification tiering (in-app only — no email channel; batching prevents per-event noise):
A single student flag routes to the instructor/TA review queue with a standard in-app notification.
Auto-pause triggers an immediate, elevated-priority in-app notification to instructor and assigned TAs (visually distinct from routine flag notifications) — a distinct, urgent event, still delivered in-app only.
Instructors additionally get a daily in-app summary notification batching new flags and pending-review changes from the past 24 hours — sent only if there was any new activity in that window; no activity, no summary notification.
TAs can investigate and escalate with a recommendation (correct / archive / clear), but every flag resolution requires instructor sign-off. Instructor resolutions: correct (edit, keep active), archive (remove from use, retain records), or clear (dismiss, unchanged). Every resolution closes the flag and records the outcome; see §6.2 for the content-error remediation process when a resolution reveals the question was actually wrong.
ST-P09 — Student flagging. "Flag this question" is available on the active question and its feedback view, one action, non-blocking. An optional short reason may be attached (submittable blank). The student gets a brief confirmation; re-flagging the same question in the same context is idempotent (no duplicate records). Flagging changes nothing about the question itself — it only routes it to the review queue.
5. Student Workflow
5.1 Topic Practice
A student selects a Theme (chapter), views its Learning Objectives (LOs), and practices in order or jumps to a specific LO — or practices at the Theme level, where the system serves a blend across all LOs in the Theme.
Question selection (per practice turn): filter the Approved bank to questions tagged with the current LO (including multi-tagged questions); exclude questions already served in this LO within the current session (IN-Q13); pick a target difficulty from the mastery-driven progression rule (advance a tier on a correct answer, repeat the same difficulty/concept on a Common Misconception miss, step back a tier on repeated misses concentrated on Hard questions); randomly select within the remaining difficulty-matched pool (randomizing parameter values at serve time for parameterized questions). If that pool is empty, degrade gracefully in order: same difficulty but already-served-this-session (repeats allowed), then adjacent difficulty unseen, then any Approved question for the LO regardless of difficulty/repeat status. (An LO with zero Approved questions is never shown to students at all — ST-P01/ST-P02 already gate on ≥1 Approved question.) Repetition of a thin LO's questions is expected system behaviour when the bank is small, not a bug.
Multi-LO questions (tagged to more than one LO/Theme, IN-Q13) can be served under any of their tagged LOs, but a mastery update from one serving only updates the LO context it was served under — not every LO the question is tagged to. This keeps mastery scoring transparent (no single answer silently boosts several LOs at once) at the cost of requiring the concept to be independently verified in each LO context.
Feedback and Strategy A specifics: the retry offered by Strategy A is on a new question testing the same concept, not the same question again; the original correct answer and other explanations stay hidden until the retry is resolved. The original wrong answer is added to the Review Book immediately on the first miss, regardless of the retry's outcome. The retry attempt is recorded as its own, independent mastery attempt (not discounted or merged with the original). If no second Approved question exists for that concept/LO to retry with, the flow degrades to Strategy B for that instance — skip the retry step and reveal the full explanation immediately, rather than blocking on a retry that can't happen.
Feedback strategy: a Common Misconception error shows only the chosen option's explanation plus one retry (Strategy A, as above); a random/other error shows all explanations immediately (Strategy B). See §6.3 for the course-level Adaptive/A-only/B-only configuration.
Follow-up conversation — not in MVP. Feedback is one-directional only in MVP; there is no follow-up chat. Rationale: FinanceBot's original design goal was to guarantee content accuracy by keeping all served content pre-reviewed; open-ended conversational follow-up reintroduces hallucination risk even when grounded in course materials via RAG, and validating it properly needs testing time the pilot timeline doesn't have slack for. Conversational feedback is tracked as an Under Evaluation item (§10): tested internally in parallel, off the student path, and promoted to a later phase only if testing shows an acceptable accuracy rate. The AI boundary rules below are pre-specified now so reactivation doesn't require new design work later:

Not reachable — the follow-up entry point is closed during the retry gate entirely

Product framing: FinanceBot's value isn't "AI writes questions, students do them" — that description alone doesn't distinguish it from a static question bank tool. It's three things working together: (1) an instructor-facing pipeline that keeps content production fast enough that the review bottleneck (§6.2) doesn't starve the student experience; (2) a strict human-approval gate ensuring nothing unreviewed reaches a student; (3) a student-facing engine (this section) that adapts difficulty, feedback depth, and progression per student, rather than serving a static problem set. Removing any one of the three collapses the product into something materially smaller than the TLEF grant's "Personalized learning at scale" objective — which is why full scope is retained rather than trimmed (§11 covers how that scope risk is managed).
Sessions open with a prior-session summary and close with a recap that can be deferred to seed the next session. The full in-session Q&A transcript stays visible and scrollable, with additional practice requestable from any history entry.
Stretch goals: in-context hints via a delayed "Request a hint" button (points to materials, never reveals the answer); cross-session history visible to students.
ST-P01 — Theme selection & Theme-level practice. The course home lists all active Themes that have ≥1 Approved question (others hidden), each with a coverage indicator reflecting the persisted mastery profile. Each Theme has a "Practice this Theme" control (blended LO practice, no LO selection required); activating the Theme name opens its LO list. Themes with a future "Available from" date are hidden until that date; Themes archived mid-term disappear from the home view. An empty course shows a "no practice topics available yet" state.
ST-P02 — LO list. Selecting a Theme shows its ordered active LOs with ≥1 Approved question, each labelled Not attempted / In progress / Covered / Struggling. "Start practice" begins at the first uncovered LO in configured order; any LO row is directly selectable, including Covered ones (re-entry doesn't reset status). Archived LOs disappear mid-term; a Theme whose last active LO is removed is hidden.
ST-P03 — One question at a time. Exactly one question is shown: MCQ = stem + four options (A–D); True/False = statement + two options. Selecting marks a pending choice; a "Submit" control commits it (selection freely changeable until then). There is no way to advance without submitting or explicitly skipping the LO. Stems with tables or simple visuals render inline. Parameterized questions display values randomized at serve time, fixed for the attempt (never re-rolled mid-question).
ST-P04 — Immediate feedback. Feedback appears on submit without navigation; options lock for that attempt. Correct: confirmation + correct-option explanation + progression recommendation. Common Misconception selection (Adaptive/Strategy A): chosen option's explanation only, plus one retry on a new same-concept question before full reveal (or immediate full reveal if no retry question is available). Other wrong selections (Adaptive/Strategy B): all per-option explanations and the correct answer immediately. A locked strategy applies regardless of option type. A mid-session strategy change applies from the next question; the in-progress question is unaffected.
ST-P05 — Progression recommendations. When the mastery engine marks an LO Covered, an advance-to-next-LO recommendation surfaces at the next natural break (after feedback); when all LOs in a Theme are Covered, an advance-to-next-Theme recommendation surfaces (last Theme → points to session recap). Recommendations are suggestions only — the student can decline and continue. While an LO is uncovered, the system serves another question on it, targeted by the mastery pattern. Current LO/Theme and live status are visible during practice.
ST-P06 — Manual LO skip. "Skip this LO" is available on any LO during practice, recorded as "skipped after attempting" or "skipped without attempting." Skip advances to the next LO (or the LO list if last). Skipped ≠ Covered: the LO stays selectable and new attempts supersede the skipped state. Skipping carries no penalty, block, redirect, or instructor flag.
ST-P07 — Repeated-failure redirect. When failures on an LO meet the configured redirect threshold (clustered on lower-difficulty questions, appearing random or rushed — §9.2), a non-modal message points to the LO's course materials plus any instructor-attached approved external reference. It always includes an option to continue practicing (never a hard stop), never reveals the current answer, and silently records a flag on the instructor dashboard. Precedence: if the student fails only higher-difficulty questions, the system steps back to easier questions instead of redirecting.
ST-P08 — In-session transcript. Every answered question, selection, feedback, and retry stays visible in a scrollable, read-only transcript (no re-answering from history). From any entry, the student can request additional fresh practice on that question's LO. Scrolling back never loses the live question. Transcript is per-session only; mastery profile and Review Book persist across sessions.
ST-P09 — see §4.3.
ST-P10 — End-of-session summary. On session end: LOs/Themes covered, questions attempted, accuracy by LO, Review Book additions, and recommended next steps — with direct links into Review Book entries or Topic Practice per item. The student can defer the summary as the starting point for the next session. An empty session shows a minimal no-activity state. Summary data matches the persistent mastery profile (consistent with the instructor dashboard).
ST-P11 — Session-start summary. Opening a course shows a brief prior-session summary (LOs/Themes covered, mastery status, recommended starting point) before practice. First session: replaced by a welcome prompt to pick a first Theme. Deferred end-of-session items appear as the primary recommendation. Always dismissible; the student can navigate freely.
5.2 Exam Prep Mode
The student chooses midterm or final; the system assembles a practice exam matching the instructor's template (Themes, question counts, MCQ/T-F splits, point values, time limit) to simulate the real exam. Only available inside the instructor-configured availability window. No per-question feedback, no retries, no hints, no follow-up questions. At the end: overall score and breakdown by Theme (LO-level optional per instructor config), with navigation into Review Book and Topic Practice. Students can view a history of all past practice exams.
ST-X01 — Exam assembly. The "Exam Prep" entry point is visible only when ≥1 saved exam template's availability window is active (hidden otherwise). Student selects among the configured/available exams. Assembly matches the template exactly (Themes, counts per Theme, MCQ/T-F split per Theme, point values), drawing exclusively from the Approved bank (sourcing invisible to students). If a Theme's quota can't be filled, the exam assembles with what's available and the gap is recorded for the instructor — the student is never blocked. Parameterized values randomize at serve time and stay fixed for the attempt.
ST-X02 — Single-sitting conditions. One continuous attempt; no correctness indication, explanations, or answers during it. Answers can be changed before final submission but not re-attempted with feedback. The student can navigate among questions and see answered vs. unanswered; "Submit exam" warns on unanswered questions before final submission. Interruptions resume the same attempt with answers retained and no feedback leakage. If a time limit is configured, a countdown is visible throughout, a warning appears at 5 minutes remaining, and expiry auto-submits current answers. Hints and follow-up questions are disabled.
ST-X03 — Results breakdown. On submission: overall score + per-Theme breakdown (per-LO within Themes if instructor-configured). Post-scoring, every question is reviewable with correct answer and per-option explanations revealed. Weak Themes/LOs link directly into Topic Practice or Review Book. Incorrect answers auto-collect into the Review Book (ST-R01). Results persist and feed the mastery profile (via the post-exam batch pass — §9.2) and instructor analytics.
ST-X04 — Exam history. A dedicated view lists all completed practice exams (type, date, score) with access to the full breakdown and per-question review identical to the post-submission view, plus navigation into Topic Practice from any past result's Theme/LO.
5.3 Review Book
Every incorrectly answered question (Topic Practice and Exam Prep) is auto-collected into a persistent Review Book; students can also manually bookmark any question. Two entry modes: browse by Theme/LO to see where gaps concentrate, or jump straight to questions for re-practice. Default view is collapsed (Themes with entry counts; questions shown only after selecting a Theme). On re-practice, parameterized questions get fresh randomized values; conceptual questions are presented as originally shown.
ST-R01 — Auto-collection. Incorrect answers are added automatically with full context (question, selected option, correct answer, timestamp), immediately on the miss regardless of any retry outcome (§5.1). One entry per question (repeat misses update the latest-attempt context, no duplicates). Entries persist across sessions until term end or student removal. Answering correctly later never silently deletes an entry — the re-practice outcome is reflected on it.
ST-R02 — Manual bookmarking. A bookmark ("Save to Review Book") control is available on any question regardless of correctness, and toggles reversibly. Bookmarked entries are visually distinguishable from auto-collected ones; a question that is both appears once carrying both signals.
ST-R03 — Re-practice. From the Review Book, re-practice runs over selected entries or a whole Theme/LO group. Each question is a fresh attempt with full per-option feedback after submission. Re-practice attempts update the mastery profile identically to original attempts (not discounted). Removing an entry affects only the Review Book, never the underlying answer history.
ST-R04 — Fresh values on re-practice. Parameterized questions re-randomize within configured ranges per re-practice attempt (same template and concept, different values; stable within an attempt). Conceptual questions are unchanged.
ST-R05 — Organisation & sorting. Default collapsed Theme view with counts; entry-mode choice (browse by Theme/LO vs. direct question list). Re-practice ordering options: Theme/LO (default), question type, date added (newest first), difficulty (either direction), random shuffle, or manual reorder. Empty Review Book shows an encouraging empty state.
ST-R06 — Post-session missed list. The end-of-session summary lists that session's missed questions (same entries as auto-collected, not a divergent list) with direct links into re-practice, and can be deferred to seed the next session.
ST-R07 — Review → targeted practice. From any Review Book question, the student can jump into live Topic Practice for its LO (fresh Approved questions, identical behaviour to normal Topic Practice), or request more practice across the LO set tied to their misses. If no additional Approved questions exist for an LO, the system says so plainly.
5.4 How Student Features Connect
One cohesive loop in one interface: Topic Practice populates the Review Book automatically; Review Book re-practice updates mastery like any attempt and flows back into Topic Practice; the Exam Prep breakdown points back to the Themes, LOs, and Review Book entries needing attention.
6. Instructor Workflow
6.1 Course Setup & Content Management
Instructors use the same CWL-authenticated web interface as students (no separate panel or installation) and build courses themselves — the dev team does not construct courses on their behalf. Course structure is never frozen: Themes and LOs can be added, renamed, archived, merged, split, and reordered at any point in the term, and each Theme can carry an "Available from" date for progressive release. The instructor creates the course, configures the Course → Theme → LO hierarchy, sets term dates, generates the registration code, and uploads materials (PDF, Word, PPT, text, URLs). After upload, the system suggests a Theme/LO hierarchy from the content and auto-classifies materials to Themes/LOs — both fully overridable. Stretch goal: attaching vetted external references to a Theme/LO.
Hierarchy change propagation: renaming a Theme/LO doesn't affect any linkage — the ID is stable, only the display label changes. Merging two LOs automatically re-links every question tagged to either into the merged LO the moment the merge is confirmed (the originals are archived) — a system-driven bulk operation, not manual per-question work; a preview step (matching IN-L01's existing pattern for material-replacement previews) lists every question that will be re-linked before the instructor confirms. Splitting one LO into two requires the instructor to manually assign each existing question to the new LO(s) before the split can be confirmed, since the system can't infer which sub-topic a question belongs to. New/imported questions go through the normal AI-suggested-then-confirmed tagging flow (IN-Q13) and aren't affected by this.
Crucially, none of the above retroactively touches already-recorded student mastery or Review Book data — those records are pinned to the LO context that was actually served at the time (via the AttemptRecord model, §2), so they remain valid even after the active hierarchy has moved on. Only the live, currently-served question↔LO linkage changes.
IN-S01 — Course & hierarchy creation. "Create course" requires course name, course code, and term. Themes require a name (optional "Available from" date); LOs require a name; both are manually reorderable and displayed in a single expandable Course → Theme → LO view. Empty states prompt the next step. Duplicate Theme/LO names within the same scope warn inline but don't block. After materials upload, the system proposes a full hierarchy the instructor can accept, modify per item, or reject entirely.
IN-S02 — Term dates. Term Start and End dates are required before publishing; End must be after Start (inline validation). Reaching the End date automatically revokes student access (course shows inactive, is not deleted). Dates are editable post-publish with immediate effect and no impact on historical data. Individual students can be granted extensions past term end from the roster (per-student end date; course settings unaffected).
IN-S03 — Registration code. One-click generation of a unique alphanumeric code, always visible in Course Settings, copyable in one click. Regeneration immediately invalidates the old code without affecting already-enrolled students.
IN-S04 — Material upload. Accepts PDF, Word, PowerPoint, plain text, and URLs; unsupported types get an inline error naming the format. Multi-file uploads process independently (one failure doesn't block others). Uploads appear in a material list with name, format, upload date, and processing status (Processing / Ready / Failed), with retry on failure. URL ingestion fetches and processes public pages only — Canvas and other auth-required URLs are unsupported; unreachable URLs error inline with the URL identified. Materials can optionally be pre-assigned to a Theme/LO at upload (skips auto-classification but remains adjustable).
IN-S05 — Manual material assignment. Each material can be assigned to one or more Themes/LOs via the full hierarchy view with multi-select. Each Theme/LO shows its assigned materials. Unassigned materials sit in a labelled "Unassigned" section with a prompt. Reassignment/removal never deletes the material or questions already generated from it.
IN-S06 — Auto-classification. After processing, each material gets a suggested Theme/LO assignment with a confidence indicator; the instructor accepts, modifies (manual interface pre-filled with suggestions), or rejects (leaves unassigned). Low-confidence materials are marked "Unclassified" with a manual-assignment prompt.
IN-S09 — Hierarchy evolution. Inline rename (reflected immediately everywhere). Archive removes a Theme/LO from the active structure and student view while retaining its questions inactive in the bank. Merge two LOs (per the propagation rule above). Split one LO into two (per the propagation rule above). Bulk-reassign selected questions to a different Theme/LO with a count confirmation. Archiving anything with active Approved questions warns with the affected count first.
6.2 Question Bank Management
Question lifecycle — four separate concepts:
Publication state (determines whether a question can be served): Draft → Pending Review → TA/Instructor-reviewed → Approved → Paused → Archived. Only Approved is ever served to students. Paused is a formal state: a question enters it automatically via auto-pause (§4.3) or manually by instructor action, and returns to Approved once the underlying flag is resolved as "correct," or moves to Archived if resolved as "archive." Archived is reachable from any state (IN-Q07).
Review decision (the events that move a question between publication states): Agent pass / Agent flag / Agent reject, Mark as reviewed (TA or instructor — the action, distinct from the "TA/Instructor-reviewed" state it produces), Instructor approved, Instructor rejected.
Labels (overlay metadata, never gate serving on their own): Source changed, Student flagged, Convertible to parameterized, Auto-converted, Manually edited.
Flag case state (its own small state machine, decoupled from the question's publication state): Open → Escalated → Resolved: corrected / Resolved: archived / Resolved: cleared.
Every generated/imported question enters as Draft; at Draft the instructor can add missing role labels, explanations, parameterization settings, or LO associations. Instructors can edit any field (stem, options, role labels, explanations, LO/Theme tags — a question may belong to multiple LOs/Themes), approve, reject, regenerate with custom prompt guidance, or archive. Instructors and TAs can leave internal notes (timestamped, attributed, teaching-team-only) on any question in the review pipeline. Pre-pilot, the bank is pre-seeded to 3–5 Approved questions per LO. On import, the system flags questions convertible to parameterized form and proposes the parameterization (including underlying code) for instructor review and approval. The bank is browsable/filterable by Theme, LO, type, and status, plus a hierarchical Course → Theme → LO tree view with per-level question counts. Themes/LOs with no Approved questions are hidden from students.
Content QA: Before pilot launch, the instructor performs a notation/terminology consistency pass over the pre-seeded question bank to confirm generated explanations align with the course's specific conventions (e.g., variable names, formula presentation); systematic mismatches are corrected via prompt/context adjustment rather than per-question edits.
Content-error remediation: A general edit (wording, a better example, a clearer explanation) leaves prior attempt records alone — normal iteration, not evidence the earlier answers were wrong. A correctness-affecting edit — one that changes the correct answer field itself while resolving a flag as "correct" (not "clear") in IN-Q06 — is treated as a content bug and triggers remediation:
Locate affected records: only AttemptRecords pinned to the specific QuestionVersion that was wrong.
Recompute correctness for each affected attempt against the corrected answer (a factual recalculation).
Mastery: drop the corrected attempt from any rolling window it's still part of; if it's still in an affected student's current window, trigger a normal batch mastery re-evaluation (§9.2), not a special-cased one.
Review Book: remove any entry added only because of the erroneous grading.
Redirect/Struggling flags: revoke any triggered solely by the erroneous attempt.
Exam Prep, handled more conservatively: because exam results are a scored artifact the student has already seen (ST-X03), nothing is silently recalculated — the instructor is shown how many completed practice-exam attempts are affected and offered a regrade, performed only on explicit confirmation.
Notification: every affected student gets an in-app notice that a question in [LO] was corrected and their record updated (distinct from the flag-resolution notice below).
Flag resolution notifications: flag history stays attached to the QuestionVersion it was raised against, not the current version. The flagging student gets an in-app notice when their flag is resolved (corrected or cleared); students who answered an older version aren't separately notified (their records are pinned, not retroactively altered — §2). A Paused question corrected directly by the instructor returns straight to Approved without re-entering the full pipeline; a TA-suggested correction still goes through the existing TA-02 accept/modify/discard flow before it can move a Paused question back to Approved.
IN-Q01 — Import. Accepts CSV, JSON, and QTI (inline error otherwise). Import preview shows detected questions and flags parameterization candidates before confirmation. Partial imports succeed: valid questions import; failures are listed with reasons. Everything enters as Draft; Theme/LO can be assigned at import (else Draft + Unassigned). Non-MCQ/T-F questions are auto-converted and labelled "Auto-converted" for verification before approval.
IN-Q02 — Review queue. Accessible from dashboard and bank; ordered by priority: student-flagged first, then high-impact TA/Instructor-reviewed, then new questions from under-covered LOs. Each entry shows the full stem, options with role labels, per-option explanations, LO/Theme tags, and the reviewer agent's decision with reasoning. Live pending-review count; empty queue shows a confirming empty state.
IN-Q03 — Editing. On any Draft or TA/Instructor-reviewed question, editable fields: stem, option text, role labels, per-option explanations, difficulty, LO/Theme associations. Edited fields are visually distinguished from generated originals. Edits can be saved without approving. The record shows it was manually edited and when.
IN-Q04 — Approve / reject. Approve moves a question to Approved and live immediately. Reject is available only on Pending Review or TA/Instructor-reviewed questions — a Draft question hasn't been reviewed yet, so there's nothing to reject; rejecting a reviewed question returns it to Draft with an optional note. Bulk approve with a count confirmation. Status labels update immediately in queue and bank.
IN-Q05 — Agent decisions surfaced. Every Draft/Pending question shows the reviewer agent's decision (Pass / Flag / Reject) plus reasoning covering factual accuracy, calculation correctness, LO/materials alignment, distractor quality per assigned role, and overall clarity. The structure validator's per-role assessment (four roles — see §9.1) is shown alongside. The queue filters by agent decision.
IN-Q06 — Flag resolution. See Content-error remediation and Flag resolution notifications above; auto-pause threshold per §4.3.
IN-Q07 — Archiving. Any question can be archived from any state; archiving an Approved question requires confirmation (Draft/Pending doesn't). Archived questions are hidden from the default bank view (visible via an "Archived" filter), read-only, and never served. Restore returns a question to Draft, requiring re-approval.
IN-Q08 — Browse, search, filter. Default view: all non-archived questions with stem, type, Theme/LO tags, difficulty, and publication state — the state filter is strictly the six publication states (Draft / Pending Review / TA-Instructor-reviewed / Approved / Paused / Archived); "Flagged" is a separate, combinable label filter, not a state option. Combinable filters: Theme(s), LO(s), type, difficulty, publication state, and labels. Always shows the filtered result count; empty results prompt filter adjustment.
IN-Q09 — Parameterization config. Per numerical question: a panel showing detected variable slots and their ranges, each slot definable by name, min, max, and step size or allowed value set, visually linked to stem placeholders. Preview with sample randomized values before saving. Questions judged convertible carry a "Convertible to parameterized" flag. Settings save independently of approval state.
IN-Q10 — Pre-seeding. Dashboard progress indicator: Approved / TA-Instructor-reviewed counts per LO against the 3–5 target, below-threshold LOs highlighted, with one-click generation of grounded Draft questions for any thin LO. LLM-assisted conversion adds missing structure to imported Drafts → Pending Review. Existing parameterization scripts can be uploaded; the system parses, maps to a template, and presents for review before entering as parameterized Draft. Publishing below threshold triggers a warning listing thin LOs but is not blocked.
IN-Q11 — Custom-prompt generation. A generation interface with free-text prompt, target LO, question type, and difficulty controls. Uploaded materials referencable via @ mentions. Preset prompt templates populate the input for editing. Output enters as Draft with the prompt recorded.
IN-Q12 — Regeneration. "Regenerate" on any Draft/Pending question opens a prompt pre-filled with the original generation prompt (if any), editable. The original stays unchanged until explicitly replaced; new variant is shown side-by-side for comparison. Regeneration attempts are recorded with dates.
IN-Q13 — Multi-LO/Theme association. Questions can be tagged to any number of LOs and Themes (all shown as tags in the bank). A multi-tagged question can be served in any of its LOs' practice sessions but never twice within a single LO session. The system auto-suggests associations for generated/imported questions; the instructor accepts, modifies, or rejects before saving.
6.3 Course Configuration
IN-S07 — Exam templates. Separate midterm and final templates in Course Settings. Required fields per template: covered Themes, questions per Theme, MCQ/T-F split per Theme, point values (per question or per Theme), time limit. Each template has an availability window (start/end dates) gating the student-facing entry point. If Approved-question supply for a Theme falls short of the template, an inline warning identifies the Theme and shortfall (saving still allowed). Data is exportable for pilot evaluation and retained post-expiry for research, subject to the consent model in §4.1.

IN-A02 — Answer distributions. Per question: counts and percentages per option. Disproportionately high Common Misconception selection is visually highlighted. Distribution is viewable aggregated at Theme and LO level. "Insufficient data" below the attempt threshold.
IN-A03 — Engagement. IN-A05 — Post-expiry data & research. After term end: instructors retain read-only access to all analytics, histories, mastery profiles, and engagement data; students can no longer access or submit. Research use requires UBC REB approval and the CWL PIA; research exports replace PUIDs with non-identifiable research IDs and only include students who gave the optional research-export consent (§4.1); individual identities are never shared outside the teaching team beyond that.
IN-A06 — Struggle flags (renamed from "Distillation flags"). 9. System Behaviours
9.1 Question Serving & Generation Pipeline
Only Approved questions are ever served; Themes/LOs with no Approved questions are hidden from students. Generation is always decoupled from serving: no student action ever triggers real-time unreviewed content reaching a student, and a thin bank results in a hidden Theme/LO — never a fallback to an unreviewed question. (This diverges from the PSD's current "RAG generation fallback" language and needs reconciling with Saurav/Stephen — see §11.) Parameterized questions get randomized values at serve time. Routing is invisible to students; a multi-LO question can legitimately surface under different LOs across sessions (mastery updates only the served LO context, §5.1).
Generation runs a three-agent pipeline: generator produces the question; structure validator validates the option roles (Correct Answer, Common Misconception, Partially Correct, Clearly Wrong / Implausible — four roles, merged down from an earlier five-role draft that didn't match the 4-option MCQ structure); reviewer verifies the answer and reasoning. The system can be configured to generate the Correct Answer and Common Misconception via separate model calls for quality. Instructors can edit or manually author any option, including the Common Misconception, at approval.
Only two roles carry behavioral logic anywhere in the system — Common Misconception (drives Strategy A's retry gate) and Clearly Wrong/Implausible (drives the disengaged-failure/struggle signal, §9.3). True/False does not get a separate taxonomy: a T/F distractor is, by design, a plausible-sounding wrong statement, so its single incorrect option always maps to Common Misconception; the Clearly Wrong/Implausible role simply doesn't apply to a two-way choice.
Instructor materials containing embedded practice questions are auto-detected and extracted, then processed through the same conversion pipeline as imports.
Review-queue prioritization: student-flagged questions first, then questions frequently answered incorrectly, then unreviewed new questions from under-covered LOs. The instructor is notified in-app (per §4.3's tiering) when pending approvals accumulate past an instructor-set threshold.
9.2 Mastery Engine
Purpose: drives two things — the student-facing adaptive navigation (what to serve next, when to recommend advancing, when to trigger a material redirect) and the instructor-facing diagnostic signal (which LOs are broadly weak, which students need outreach). Outputs a four-state label per LO (Not attempted / In progress / Covered / Struggling), never a numeric score shown to students.
Two-layer design, built around the bank actually available rather than fixed rules:
Layer 1 — statistics (cheap, deterministic, computed on every attempt): a rolling window of the most recent 10 attempts per (student, LO) feeds attempt count, accuracy, and the distribution of selected option roles. Pure bookkeeping — no judgment.
Layer 2 — LLM judgment (the mastery evaluator, §2/AD-07's fourth pipeline step): given the student's full attempt history for that LO plus the Layer-1 statistics, returns: (1) status (Covered / In progress / Struggling), (2) a one-line rationale surfaced on the instructor dashboard, (3) whether more of a particular question type would help — constrained by what's actually in the bank, so a thin bank never blocks a student on a question tier that doesn't exist.
When the evaluator runs: not on every attempt. Default cadence is every 5 attempts on a given LO, batched, running asynchronously via the job queue (§2) so it never blocks real-time feedback. A fast-track exception bypasses the batching when Layer-1 stats show a strong disengaged pattern (e.g., two consecutive Clearly Wrong/Implausible selections), triggering evaluation immediately.
Specific rules:
Skipped LOs and Theme coverage: a Theme is Covered once all its non-skipped LOs are Covered; skipped LOs neither block nor count against Theme completion. The dashboard separately notes when completion includes skipped LOs.
Review Book attempts: full weight in the same rolling window, tagged mode: Review Book on the AttemptRecord.
Exam Prep attempts: recorded, processed in a separate post-exam batch pass, not the live cadence. A miss adds an independent "exam-verified" qualifier to that LO's status rather than overwriting the practice-derived status.
Multi-LO questions: update only the LO context under which the question was actually served, never all of a question's tagged LOs.
Difficulty recalibration stability: difficulty is initially LLM-scored, instructor-overridable, refined over time from performance data — but recalibration only changes future servings' difficulty. Each AttemptRecord pins the difficulty tier at serve time (via the QuestionVersion snapshot), so recalibration never retroactively alters a past mastery judgment.
Regression: Covered can regress to In Progress (not directly Struggling) on a subsequent miss on a harder item — "needs reinforcement," not a reset. Struggling is reached only via the disengaged-failure/struggle-signal path (§9.3), a separate trigger.
9.3 Struggle & Distillation Signal Detection
The system watches for extraction-like patterns: rapid sequential answer switching, persistent Clearly Wrong/Implausible selection, high question volume with no mastery progression. The "no progression" threshold is evaluated relative to that LO's actual Approved-bank size, not a fixed global attempt count — a student honestly working through a thin bank naturally shows more repetition than one working a well-stocked LO. Detection changes nothing for the student in real time. The student is surfaced to the instructor as a struggle flag (IN-A06), and because these signals genuinely overlap with honest difficulty, the system never takes automated action against a flagged student — interpretation is left entirely to the instructor.

11. Delivery Risks
Scope risk. §10's MVP scope is deliberately broad (§5.1's product-framing note explains why it isn't trimmed) — creating real risk that not everything is built and hardened by the September pilot start. Mitigation: the Phasing Document sequences build order against the TLEF milestones so that, if development runs behind, later-sequenced items (exam-prep polish, TA workflows, advanced analytics, admin rollout tooling) can slip without blocking the core Topic Practice + Review Book loop the pilot actually depends on.
Core-feature testing risk. The mastery engine, feedback strategy, and generation pipeline are simultaneously the features most central to the product's value and the ones most in need of testing time before real students see them. If early testing shows any underperforming, fixing them competes directly with the September deadline. Mitigation: the Aug–Sep internal-testing window is treated as a hard quality gate; the existing reviewer-toggle and pre-seeding-threshold controls serve as a fallback to keep unreliable content away from students while fixes continue.
Dependency risk. The RAG toolkit and its packages are pre-1.0 (§2); breaking changes during integration could consume schedule buffer. Mitigation: pin exact versions and budget explicit compatibility-shim time.
Compliance timeline risk. The CWL PIA/DAR (§4.1) is on the critical path to any student-facing launch and runs on UBC IAM's own review timeline, outside the project team's control. Mitigation: initiate the process in parallel with development, as early as possible.
Scaling risk. WT2 expansion (~200 combined students) exercises concurrency the pilot itself won't. Mitigation: load-test against the §2 System Requirements concurrency targets before WT2 enrollment opens, not after.
Open cross-team item. §9.1 documents the PRD's serving policy as strict Approved-only, which currently diverges from the PSD's "RAG generation fallback" language — needs reconciling with Saurav/Stephen so the two documents state the same behaviour.
